# 数据卷与网络

> [!info] 所属：[[Docker MOC]] · 上一篇 [[容器操作]] · 下一篇 [[Docker Compose]]

## 核心概念

容器天生** ephemeral（用完即弃）**——删了数据就没了，且容器间默认隔离。要让数据**持久化**、让容器间**能通信**，就需要 Volume（数据卷）和 Network（网络）。这是把容器用于真实业务的关键。

### 数据持久化的两种方式
| 方式 | 命令形式 | 特点 |
| --- | --- | --- |
| **Volume（命名卷）** | `-v mydata:/app/data` | Docker 管理，与宿主解耦，**推荐**，跨平台、易备份迁移 |
| **Bind Mount（绑定挂载）** | `-v /宿主/路径:/app/data` | 直接挂宿主目录，开发热重载方便，但依赖宿主路径 |

> 生产用 **Volume**（Docker 托管、不依赖宿主目录结构）；**开发**用 Bind Mount 挂源码实现热重载。

### 网络模式
容器默认在 Docker 管理的桥接网络里。要让容器间互相访问，把它们放到**同一自定义网络**——这样可以用**容器名作主机名**直接通信（DNS 自动解析）。
- **bridge**（默认）：单机容器互通。
- **host**：容器直接用宿主网络（无隔离，端口不冲突）。
- **none**：无网络。
- 自定义 bridge：**推荐**，容器间可用名字互访，比默认 bridge 更安全隔离。

### 端口映射 vs 容器互联
- `-p 宿主:容器`：把容器端口**暴露给外部/宿主**访问（外部 → 容器）。
- 同一自定义网络内：容器间**直接用容器名:端口**访问，**无需** `-p`（容器 ↔ 容器）。

## 代码示例（命令）

```bash
# ============ 数据卷 ============
# 1) 命名 Volume（推荐）：Docker 管理，删容器数据不丢
docker volume create pgdata
docker run -d --name db -v pgdata:/var/lib/postgresql/data postgres:16
# 即便 docker rm db，pgdata 卷仍在，数据保留
docker volume ls                    # 列出所有卷
docker volume rm pgdata             # 显式删卷（数据才真没）

# 2) Bind Mount：挂宿主目录（开发热重载）
docker run -d --name dev \
  -v $(pwd)/src:/app/src \          # 宿主 ./src ↔ 容器 /app/src
  -v $(pwd)/src:/app/src:ro \       # :ro 只读挂载
  myapp
# 在宿主编辑 src，容器内立即生效（无需重建镜像）

# ============ 网络 ============
# 1) 创建自定义网络
docker network create mynet

# 2) 容器加入同一网络 —— 可用「容器名」互访
docker run -d --name db --network mynet -e POSTGRES_PASSWORD=secret postgres:16
docker run -d --name web --network mynet -e DB_HOST=db myapp
# web 容器内用主机名 "db" 即可连数据库（DNS 自动解析），无需 -p 暴露 db

docker network ls
docker network inspect mynet        # 看网络详情与成员

# 3) 端口映射（对外暴露）
docker run -d -p 8080:80 --network mynet web   # 外部经 localhost:8080 访问 web
# 注意：db 不需要 -p，它只被 web 内部访问（更安全）

# ============ 实战：开发时源码热重载 ============
docker run -d --name api \
  -v $(pwd):/app \                  # 挂载源码
  -p 8000:8000 \
  myapi:dev                         # 启动命令配 --reload（如 uvicorn --reload）
```

## ⚠️ 易错点 / 面试要点

- **删容器不删 Volume**：`docker rm db` 删的是容器，命名 volume 的数据**保留**（防误删）。要连同删加 `-v`（`docker rm -v db`）。`system prune` 默认也不删命名卷（加 `--volumes` 才删）。
- **生产用 Volume，开发用 Bind Mount**：Volume 不依赖宿主路径、跨平台、好备份；Bind Mount 依赖宿主绝对路径，换机器易坏，但开发改代码即时生效。
- **容器间通信用自定义网络 + 容器名**：默认 bridge 网络里容器只能用 IP 互访（IP 会变）；自定义网络自动 DNS，用容器名稳定访问。
- **别给内部服务加 `-p`**：只有需要**对外暴露**的服务才 `-p`（如 web）。数据库等内部服务不映射端口更安全（仅同网络容器可访问）。
- **权限问题**：Bind Mount 挂的文件，容器内进程的 uid 若与宿主文件 owner 不一致，可能读写失败。常见于挂载 config 文件。
- **Volume vs 容器可写层**：重要数据必须放 Volume，别依赖容器可写层——它随容器删除而消失，且写入性能差。
- **`host` 网络移除隔离**：`--network host` 容器与宿主共享网络栈，端口直接用宿主的，失去隔离，仅特殊场景用。
