# 测试基础

> [!info] 广度概览 · 回到 [[🗺️ 知识地图 MOC]]

## 是什么

测试（Testing）是**用自动化的方式验证程序 behaves as expected**。它的目标不是「证明没有 bug」，而是「在用户发现之前尽早暴露问题」，并为后续重构提供安全网。

## 为什么重要

- 没有测试的代码，你不敢改；不敢改的代码就成了遗产。
- 测试是最好的**活文档**：读测试就知道函数该有什么行为。
- 自动化测试是 CI/CD（[[DevOps 概览]]）和敏捷迭代的前提。

## 核心要点

### 测试金字塔
```
        /\
       /E2E\        少而慢（真实用户流程，如点完整登录）
      /------\
     / 集成测试 \    中等（多模块协作，如 API + 数据库）
    /----------\
   /  单元测试   \   多而快（单个函数/类，毫秒级）
  /--------------\
```
> 底层多、顶层少：单元测试便宜且稳定，应占绝大多数。

### 黑盒 vs 白盒
- **黑盒**：只看输入输出，不关心内部实现（多数 E2E、API 测试）。
- **白盒**：基于内部结构设计用例（路径覆盖、分支覆盖）。

### TDD（测试驱动开发）
红-绿-重构循环：先写失败的测试（红）→ 写最少代码让它过（绿）→ 优化结构（重构）。
- 好处：逼你想清楚接口、天然 100% 覆盖、设计更解耦。

### 关键概念
- **断言（assertion）**：期望某个条件成立，不成立则失败。
- **Mock / Stub**：用假对象替换真实依赖（数据库、网络），隔离被测单元。
- **Fixture**：测试前置的准备数据/环境。

### 覆盖率不是万能
- 覆盖率高 ≠ 没 bug：它只说明「跑过」，不说明「断言充分」。
- 盲目追 100% 覆盖率反而催生无意义测试。

> [!tip] 心态
> 测试**行为**，不测**实现**。否则一重构测试就全红，反而成负担。
