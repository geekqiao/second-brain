# 高级操作

> [!info] 所属：[[Git MOC]] · 上一篇 [[远程协作]] · 下一篇 [[排错与最佳实践]]

## 核心概念

这些命令是「救火」与「精修」的工具：临时搁置改动、搬运单个提交、撤销错误、找回丢失的提交、整理历史。掌握它们，Git 就从「能用」变成「可控」。

### stash（暂存搁置）
手头改到一半、突然要切分支修 bug：`stash` 把工作区改动**临时存起来**，工作区变干净；回来后 `pop` 恢复。

### cherry-pick（摘樱桃）
把**某一个提交**单独搬到当前分支。适合「feature 分支里有个 bugfix，main 也急需」的场景。

### reset（重置，改写历史）
移动当前分支指针，有三种模式：
| 模式 | 工作区 | 暂存区 | 效果 |
| --- | --- | --- | --- |
| `--soft` | 保留 | 保留 | 只移指针，改动全在暂存区（可重新 commit） |
| `--mixed`（默认） | 保留 | 清空 | 改动退回工作区（未暂存） |
| `--hard` | **清空** | **清空** | 彻底丢弃（危险，不可逆于工作区） |

### revert（反向提交，安全撤销）
生成一个**新提交**来「抵消」指定提交。不改写历史 → **适合已推送的公共分支**。

### tag（标签）
给某个提交打**语义化标记**（如 `v1.0.0`），常用于版本发布。`-a` 附注标签带信息。

### reflog（后悔药）
记录 HEAD 的**每一次移动**（包括被 reset 丢弃的提交）。即使「删掉」了分支/提交，reflog 里通常还能找回。**Git 几乎没有真正的「永久丢失」**（除非 GC）。

## 代码示例（命令）

```bash
# ============ stash：临时搁置 ============
git stash                     # 搁置当前改动（工作区变干净）
git stash list                # 看有哪些搁置
git stash pop                 # 恢复最近一次搁置并删除它
git stash apply               # 恢复但保留搁置（不删）
git stash drop                # 删除某次搁置

# ============ cherry-pick：搬运单个提交 ============
git switch hotfix
git cherry-pick a1b2c3d       # 把 a1b2c3d 这次提交应用到当前分支
git cherry-pick A..B          # 搬运一段提交（不含 A，含 B）

# ============ reset：重置分支指针 ============
git reset --soft HEAD~1       # 撤销最近一次提交，改动留在暂存区（可改信息重提）
git reset HEAD~1              # 默认 mixed：撤销提交，改动退回工作区
git reset --hard HEAD~1       # ⚠️ 彻底丢弃最近一次提交+工作区改动
git reset --hard origin/main  # 本地 main 强制与远程一致（丢弃本地所有改动）

# ============ revert：安全撤销（公共分支用） ============
git revert a1b2c3d            # 生成一个反向提交，抵消 a1b2c3d
git revert HEAD               # 撤销最近一次提交
# → 推送后历史里会有「撤销了 xxx」的记录，不改写历史

# ============ tag：版本标记 ============
git tag v1.0.0                       # 轻量标签
git tag -a v1.0.0 -m "首个正式版"    # 附注标签（推荐，带信息+签名）
git tag                              # 列出标签
git push origin v1.0.0               # 推送单个标签（tag 不会随 push 自动推送）
git push --tags                      # 推送所有标签

# ============ reflog：后悔药（找回丢失的提交） ============
git reflog                    # HEAD 的移动历史
# a1b2c3d HEAD@{0}: reset: moving to HEAD~1
# e5f6g7h HEAD@{1}: commit: 重要的提交  ← 被 reset 丢了，但 hash 还在
git switch -c recovered e5f6g7h   # 用那个 hash 建分支，提交就回来了
```

## ⚠️ 易错点 / 面试要点

- **`reset --hard` 是核武器**：它会**永久丢弃**工作区和暂存区的未提交改动（除了去 reflog 碰运气）。用前务必 `git status` 确认没有未保存的重要改动。
- **reset vs revert 的核心区别**：`reset` **改写历史**（只适合本地未推送）；`revert` **新增反向提交**（不改写历史，适合已推送的公共分支）。面试高频。
- **`reset --soft` 重新组织提交**：把几次碎提交合成一次：`reset --soft HEAD~3` → 三个提交退回暂存区 → 一次 `commit`。
- **reflog 是救命稻草**：误删分支、误 reset、误 rebase 后，`git reflog` 能看到之前的 HEAD 位置，用对应 hash 恢复。**只要没被 GC，都能找回。**
- **tag 默认不推送**：`git push` 不带 tag，要 `git push --tags` 或 `git push origin v1.0.0`。CI/部署常按 tag 触发。
- **cherry-pick 可能冲突**：搬来的提交依赖当前分支没有的改动时冲突，解决后 `--continue`。
- **`git clean -fd`**：删除**未跟踪**的文件/目录（gitignore 之外的临时文件）。同样危险，先用 `-n`（dry run 预览）。
- **recover 时机**：reflog 记录默认保留约 90 天（可达的）或 30 天（不可达的），过了才被 GC。尽快找回。
