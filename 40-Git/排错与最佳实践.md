# 排错与最佳实践

> [!info] 所属：[[Git MOC]] · 上一篇 [[高级操作]]

## 核心概念

命令会用了，真正的功力在于**出问题时能自救**，以及**用规范流程避免问题**。这一篇汇总常见翻车场景的解法，和让团队协作顺畅的最佳实践。

### 常见「翻车」与解法
| 场景 | 解法 |
| --- | --- |
| 刚提交的 commit 信息写错了 | `git commit --amend`（未推送时） |
| 漏 add 了文件 | `git add 漏的文件 && git commit --amend --no-edit` |
| 想撤销最近一次提交（未推送） | `git reset --soft HEAD~1`（保留改动） |
| 想撤销已推送的提交 | `git revert <hash>`（不改写历史） |
| 误删了分支 / reset 错了 | `git reflog` 找回 hash → `git switch -c 哈希` |
| `pull` 产生奇怪的合并提交 | 改用 `git pull --rebase`，或设全局 `pull.rebase true` |
| push 被拒（non-fast-forward） | 先 `git pull --rebase` 再 push |
| 想看某行代码是谁写的 | `git blame <file>` |
| 想丢弃所有本地改动回到远程状态 | `git fetch && git reset --hard origin/main`（⚠️ 丢弃未提交改动） |
| 冲突搞不定 | `git merge --abort`（merge）或 `git rebase --abort`（rebase）退回安全状态 |

### 提交信息规范（Conventional Commits）
```
<type>(<scope>): <subject>

<body>
```
- type：`feat`（新功能）/ `fix`（修 bug）/ `docs` / `style` / `refactor` / `test` / `chore`。
- 例：`feat(auth): 添加邮箱登录`、`fix(api): 修复空指针异常`。
- 好处：历史可读、可自动生成 changelog、CI 可按 type 触发发布。

### 分支策略（团队约定）
- **GitHub Flow**（最简）：main 可部署 + feature 分支 + PR。适合持续部署的小团队。
- **Git Flow**（重型）：main（生产）+ develop（开发）+ feature/release/hotfix 分支。适合有明确发布周期的项目。
- **Trunk-based**（现代）：大家都往 main 提交，短命分支，强依赖 CI。大厂主流。

### 提交粒度
- **一次提交一件事**：聚焦单一目的（一个功能 / 一个修复），方便 review 与回滚。
- **小步频繁提交**：比攒一大坨再提交更安全、冲突更小。
- **提交前自查**：`git diff --staged` 复查、确保能编译、不提交调试代码/敏感信息。

### .gitignore 与权限
- 提前配好 `.gitignore`（见 [[基础概念与配置]]）。
- 文件权限变更也会被 Git 记录，跨平台协作可 `git config core.fileMode false` 忽略权限变化。

## 代码示例（命令与配置）

```bash
# ============ 全局推荐配置（每台机器一次）============
git config --global pull.rebase true       # pull 时用 rebase，避免无意义合并提交
git config --global push.default current    # 只推当前分支
git config --global core.fileMode false     # 忽略文件权限变化（跨平台协作）
git config --global rerere.enabled true     # 记住冲突解决，下次相同冲突自动套用

# ============ 常见自救 ============
# 1) 提交信息写错 / 漏文件（未推送）
git commit --amend -m "feat: 正确的信息"

# 2) 撤销最近一次提交但保留改动（未推送）
git reset --soft HEAD~1

# 3) 撤销已推送的提交（团队分支）
git revert <hash>
git push

# 4) 找回误删的提交
git reflog
git switch -c recovered <hash>

# 5) 安全丢弃本地所有改动，对齐远程
git fetch origin
git reset --hard origin/main        # ⚠️ 本地未推送的改动会丢！

# 6) 合并/变基冲突想放弃
git merge --abort       # 或 git rebase --abort

# 7) 看谁写了某行
git blame file.py -L 10,20

# 8) 搜索历史里某段代码是何时引入的
git log -S "someFunction" --oneline    # 找添加/删除该字符串的提交

# ============ Conventional Commits 示例 ============
git commit -m "feat(auth): 添加邮箱验证码登录"
git commit -m "fix(cart): 修复结算时数量不更新的问题"
git commit -m "docs: 补充 README 部署说明"
git commit -m "refactor: 抽取重复的日期处理逻辑"
```

## ⚠️ 易错点 / 面试要点

- **「已推送到公共分支」是分水岭**：未推送可随便 reset/amend/rebase；**已推送就别改写历史**，用 revert。这是 Git 协作第一守则。
- **永远有 reflog**：心里慌之前先 `git reflog`，大概率能找回。Git 对已提交的内容几乎不真删。
- **`--amend` 只在本地用**：推过的提交 amend 后再 push 需要 force，会打乱协作者。
- **提交规范影响团队效率**：统一 Conventional Commits 让历史可读、能自动发版（semantic-release）。从个人项目就养成习惯。
- **大文件别进 Git**：二进制、视频、数据集会让 `.git` 膨胀且不可逆。用 Git LFS 或外部存储。
- **`git pull` 默认 merge 易产生噪音**：设 `pull.rebase true` 或每次 `git pull --rebase`，历史更干净。
- **定期清理已合并分支**：`git fetch --prune` 清理本地对应远程已删的分支。
- **备份靠远程**：本地仓库坏了不可怕，只要 push 到远程就有完整副本。**重要改动及时 push。**
