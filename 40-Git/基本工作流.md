# 基本工作流

> [!info] 所属：[[Git MOC]] · 上一篇 [[基础概念与配置]] · 下一篇 [[分支与合并]]

## 核心概念

90% 的日常 Git 就是这个循环：**改文件 → `status` 看状态 → `add` 暂存 → `commit` 提交 → `log` 查看**。把它练成肌肉记忆，Git 就不再可怕。

### 日常四步循环
1. **`git status`**：看哪些文件改了、哪些已暂存。
2. **`git add`**：把改动放进暂存区（可全加 `.`，也可挑文件）。
3. **`git commit -m "信息"`**：固化暂存区为一个提交。
4. **`git log`**：查看历史。

### 查看差异 diff
- `git diff`：**工作区 vs 暂存区**（还没 add 的改动）。
- `git diff --staged`（`--cached`）：**暂存区 vs 最近提交**（add 了但没 commit 的改动）。
- 提交前看一眼 diff，是防止提交垃圾的好习惯。

### 信息查看
- `git status -s`：精简状态。
- `git log --oneline --graph --all`：一行一提交 + 图形化分支（最实用的 log 形态）。
- `git show <hash>`：看某次提交的详细改动。

### 暂存的精细控制
- `git add <file>`：单文件。
- `git add .`：当前目录全部（含新建）。
- `git add -p`：**交互式逐块挑选**（强烈推荐，让提交更聚焦）。
- `git restore <file>`：丢弃工作区改动（未 add 的）。
- `git restore --staged <file>`：把已 add 的撤回暂存区（不丢改动）。

## 代码示例（命令）

```bash
# 完整日常循环
git status                     # 1. 看状态
git diff                       # 2. 看未暂存改动
git add .                      # 3. 暂存全部（或 git add 具体文件）
git diff --staged              # 4. 复查将提交的内容（好习惯）
git commit -m "feat: 添加登录页" # 5. 提交

# 交互式暂存（让提交更干净）
git add -p                     # 逐块选择：y 接受 / n 跳过 / s 拆更细

# 查看历史
git log --oneline --graph --all -n 20   # 图形化最近 20 条
git log --stat                 # 每次提交改了哪些文件
git show a1b2c3d               # 看某次提交的完整 diff
git blame file.py              # 逐行看是谁、哪次提交写的（追责/溯源）

# 撤销操作
git restore file.py            # 丢弃工作区未暂存的改动（回到上次 add/commit）
git restore --staged file.py   # 把已 add 的撤出暂存区（改动仍在工作区）
git commit --amend             # 修补最近一次提交（改信息或补漏加的文件）
                                   # ⚠️ 已推送的提交尽量别 amend

# 修改最近一次提交（补文件 + 改信息）
git add forgotten.py
git commit --amend --no-edit   # 并入上次提交，不改信息
```

## ⚠️ 易错点 / 面试要点

- **`git add .` 会把所有改动一锅端**：包括调试代码、临时文件。提交前用 `git status` + `git diff --staged` 复查；或用 `git add -p` 精细挑选，让每个提交聚焦一件事。
- **`commit --amend` 的风险**：它会**改写历史**（生成新提交替换旧的）。已推送到共享分支的提交别 amend，否则别人 pull 会冲突。仅限本地未推送的提交。
- **提交信息要规范**：好的信息是 `feat: 添加用户登录`，不是 `update` / `fix bug`。规范格式（Conventional Commits）让历史可读、可自动生成 changelog。
- **一次提交一件事**：把无关的改动拆成多个提交，方便 review、回滚、定位。别把「重构 + 新功能 + 改 typo」塞进一次提交。
- **`git diff` vs `git diff --staged` 的区别**：前者看未暂存的，后者看已暂存未提交的。搞混会漏看改动。
- **`.DS_Store` / `node_modules` 混进提交**：靠 `.gitignore` 提前拦截（见 [[基础概念与配置]]）。
- **`git restore`（新）vs `git checkout`（旧）**：现代 Git 用 `restore` 撤销改动、`switch` 切分支，语义比 `checkout` 清晰。
