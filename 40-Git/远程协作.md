# 远程协作

> [!info] 所属：[[Git MOC]] · 上一篇 [[分支与合并]] · 下一篇 [[高级操作]]

## 核心概念

本地仓库只是半个 Git，真正的协作发生在**远程仓库（remote）**——GitHub/GitLab/Gitea 上的中心。`push`/`pull` 同步代码，**Pull Request（PR）** 是团队 review 与合并的标准流程。

### 远程（remote）概念
- 一个本地仓库可关联多个远程，默认名为 `origin`。
- `git remote -v` 查看远程地址。
- 远程分支形如 `origin/main`，是**远程状态的本地缓存**（不会自动更新，要 fetch）。

### 同步四件套
| 命令 | 方向 | 作用 |
| --- | --- | --- |
| `git fetch` | 远程→本地（不合并） | 拉取远程更新到 `origin/*`，**不动你的工作区** |
| `git pull` | 远程→工作区 | `fetch` + `merge`（拉取并合并到当前分支） |
| `git push` | 本地→远程 | 推送本地提交到远程 |
| `git push -u origin feature` | / | 首次推送并建立**跟踪关系** |

> `fetch` vs `pull`：`fetch` 只下载不合并（安全，先看再决定）；`pull` = fetch + merge（方便但可能引入冲突）。养成先 `fetch` 看 `log` 再决定的习惯。

### 跟踪分支（upstream）
- 本地分支可「跟踪」远程分支，之后 `git pull`/`push` 不用写全名。
- `git push -u origin feature` 建立跟踪（`-u` = `--set-upstream`）。

### Pull Request（PR）/ Merge Request（MR）
**不是 Git 本身的命令，而是平台（GitHub/GitLab）功能**。流程：
1. 在自己的分支上开发并推送到远程。
2. 在平台上发起 PR：「请把我的 feature 分支合并进 main」。
3. 团队 **Code Review**、讨论、改代码。
4. 通过后由维护者合并（平台提供 merge/rebase/squash 三种策略）。

### Fork 工作流（开源贡献）
贡献别人的仓库：Fork（复制到自己账号）→ clone 自己的 fork → 建分支开发 → push 回自己的 fork → 向原仓库发 PR。

## 代码示例（命令）

```bash
# 远程管理
git remote -v                          # 查看远程
git remote add origin <url>            # 关联远程
git remote add upstream <url>          # 加第二个远程（fork 工作流的原仓库）
git remote set-url origin <new-url>    # 改地址

# 日常同步
git fetch origin                       # 拉取远程更新（不动工作区）
git log origin/main --oneline          # 看远程 main 的新提交
git switch main && git pull            # 拉取并合并到本地 main（已跟踪时）
git push                               # 推送（已跟踪时直接 push）

# 首次推送新分支并跟踪
git switch -c feature/login
git push -u origin feature/login       # -u 建立跟踪，以后直接 push/pull

# 推送本地标签（见 [[高级操作]]）
git push origin v1.0.0
git push --tags

# Fork 工作流（开源贡献）
git clone <你的fork地址>
cd repo
git remote add upstream <原仓库地址>    # 加原仓库为 upstream
git switch -c fix-typo
# ... 改代码、提交 ...
git push -u origin fix-typo            # 推到自己的 fork
# 然后到 GitHub 网页上点 "Compare & pull request" 发 PR

# 同步 fork 与原仓库
git fetch upstream
git switch main
git merge upstream/main                # 把原仓库的更新并进来
git push                               # 推回自己的 fork
```

## ⚠️ 易错点 / 面试要点

- **`fetch` 安全、`pull` 图快**：不确定时用 `fetch` + 看 `log` 再 merge，避免 `pull` 在脏工作区上制造意外合并。
- **`push` 被拒（non-fast-forward）**：远程有你没有的提交。先 `git pull`（或 `fetch` + `rebase`）合并/变基后再 push。别盲目 `--force`。
- **`--force` 慎用**：覆盖远程历史，会冲掉别人的提交。需要时用 `--force-with-lease`（带保护）。
- **`origin/main` 是缓存**：它是远程分支的**本地快照**，不会自动更新。要看真实远程状态先 `git fetch`。
- **PR 不是 Git 命令**：它是 GitHub/GitLab 平台功能，本地 `git` 里没有 `pr` 命令（用 `gh` CLI 可命令行操作 PR）。
- **提交前先 pull**：长时间没同步就 push，大概率冲突。开工前、提交前各 `pull` 一次，减少大冲突。
- **分支名别用中文/空格**：用 `feature/login`、`fix/typo` 这种英文短横线，命令行和 URL 都友好。
- **HTTPS vs SSH**：HTTPS 每次要认证（可用 credential helper 缓存）；SSH 配一次密钥长期免密（见 [[网络磁盘与用户]]）。推荐 SSH。
