# 分支与合并

> [!info] 所属：[[Git MOC]] · 上一篇 [[基本工作流]] · 下一篇 [[远程协作]]

## 核心概念

分支是 Git 最强大的特性——它让你**在不影响主线的情况下并行开发**，且极其轻量（只是个指针）。合并（merge）与变基（rebase）是整合分支的两种方式，理解它们的区别是 Git 进阶的关键。

### 分支是什么
- 分支 = 指向某个提交的**可动指针**。`HEAD` 是特殊指针，指向「当前在哪个分支」。
- 创建分支几乎不耗资源（不复制文件），所以**大胆用分支**。

### 基本分支操作（现代用 switch/restore）
- `git switch -c feature`：建并切到新分支。
- `git switch main`：切回 main。
- `git branch`：列出分支；`-d` 删除已合并的；`-D` 强制删。

### 合并 merge
把另一分支的改动并入当前分支。
- **Fast-forward（快进）**：当前分支没有新提交时，直接把指针前移，历史是直线。
- **三方合并**：两分支都有新提交时，生成一个**合并提交（merge commit）**，保留两条历史线。

### 变基 rebase
把当前分支的提交「**挪到**」目标分支顶端重新应用，得到**线性历史**（无合并提交）。
- `git rebase main`：把 feature 的提交移到 main 最新提交之上。
- **代价**：改写了 feature 的历史（提交 hash 变了）。

### merge vs rebase 怎么选
| 维度 | merge | rebase |
| --- | --- | --- |
| 历史 | 保留真实分叉（有合并提交） | 线性、干净 |
| 安全 | 不改写历史，安全 | 改写历史，**已推送的别 rebase** |
| 口诀 | 「合并别人」用 merge | 「整理自己」用 rebase |

> 黄金法则：**rebase 只用于本地未推送的分支**，绝不 rebase 已共享的分支（会让别人历史错乱）。

### 冲突（Conflict）
两分支改了**同一文件的同一区域**，Git 无法自动合并 → 冲突。需手动编辑选择保留哪部分，解决后 `add` + `commit`（或 rebase 下 `--continue`）。

## 代码示例（命令）

```bash
# 创建与切换（现代 switch 命令）
git switch -c feature/login     # 建新分支并切过去
git switch main                 # 切回 main
git branch -a                   # 列出所有分支（含远程）
git branch -d feature/login     # 删除已合并分支
git branch -D feature/login     # 强制删除（未合并也删）

# 合并：在 main 上合并 feature
git switch main
git pull                        # 先确保 main 最新
git merge feature/login         # 合并；可能产生合并提交或快进
git branch -d feature/login     # 合并后删除分支

# 变基：在 feature 上，把它的提交挪到 main 顶端
git switch feature/login
git rebase main                 # feature 的提交重新应用到最新 main 之上
# rebase 后若已推过远程，需要 git push --force-with-lease

# 解决冲突
git merge feature               # 提示 CONFLICT
# 编辑器里冲突标记：
# <<<<<<< HEAD
# 当前分支的内容
# =======
# 被合并分支的内容
# >>>>>>> feature
# → 手动改成最终结果，删除标记，保存
git add 冲突文件                # 标记已解决
git commit                     # merge 方式：完成合并提交
# 或 rebase 方式：git rebase --continue

# 查看分支图（直观）
git log --oneline --graph --all
```

## ⚠️ 易错点 / 面试要点

- **`git rebase` 黄金法则**：永远不要 rebase 已经推送到公共分支的提交。它改写历史会让协作者的仓库错乱。rebase 只整理自己本地的 feature 分支。
- **`--force` vs `--force-with-lease`**：rebase 后强推用 `--force-with-lease`（更安全，若远程有别人新提交会拒绝），别用裸 `--force`（会覆盖别人的提交）。
- **冲突要完全解决**：别留下 `<<<<<<<` 标记就提交（会导致代码语法错误）。解决后 `git add` 标记已解决。
- **删除分支是删指针，不删提交**：分支删了，其提交在 reflog 里仍能找回（见 [[高级操作]]），直到被 GC。
- **merge 保留历史 vs rebase 美化历史**：团队应统一约定。常见策略：feature 分支 rebase main 保持线性，最终 merge 进 main。
- **`switch` / `restore` 是新命令**：替代多功能易混淆的 `checkout`。`git checkout` 仍能用，但新代码推荐 `switch`（切分支）、`restore`（恢复文件）。
- **频繁小步提交 + 频繁合并 main**：feature 分支开发周期越长，合并冲突越大。定期 `rebase main` 或 `merge main` 同步，把冲突分散化。
