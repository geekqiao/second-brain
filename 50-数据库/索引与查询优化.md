# 索引与查询优化

> [!info] 所属：[[数据库 MOC]] · 上一篇 [[表设计与关系建模]] · 下一篇 [[事务与并发控制]]

## 核心概念

数据库慢，**90% 是索引问题**。索引是「用空间换时间」——额外维护一份有序结构，让查询从全表扫描 O(n) 降到 O(log n)。但索引不是银弹：建错、用错、建太多都会适得其反。这一篇讲清「何时建、建哪种、怎么验证」。

### 索引怎么工作
类似书的目录：要找某内容，先查目录定位页码，再翻过去。没有索引，只能逐页翻（全表扫描 Seq Scan）。

### PostgreSQL 索引类型
| 类型 | 适合 | 说明 |
| --- | --- | --- |
| **B-tree**（默认） | 等值、范围、排序 | 通用，`=`/`</>`/`BETWEEN`/`ORDER BY` |
| **Hash** | 仅等值 `=` | 老旧，少用 |
| **GIN** | 多值/JSONB/全文/数组 | 一个值匹配多个键（含 `@>`、tsvector） |
| **GiST** | 范围、几何、最近邻 | 地理（PostGIS）、向量近似 |
| **BRIN** | 巨大且物理有序的表 | 块级范围索引，极小，适合时序 |

### 何时建索引
- 经常出现在 `WHERE` / `JOIN ... ON` / `ORDER BY` / `GROUP BY` 的列。
- 外键列（默认不带索引）。
- 高选择性的列（值很分散，过滤掉大量行）。

### 何时**不**建 / 危害
- 写多读少的表（每次写都要更新所有索引，拖慢写入）。
- 小表（全表扫描比走索引还快）。
- 低选择性列（如「性别」，只有两值，索引没用）。
- 一张表几十个索引 → 写入灾难。

### EXPLAIN —— 看数据库怎么执行
`EXPLAIN` 看执行计划，`EXPLAIN ANALYZE` 真跑一遍看实际耗时。是调优的**核心工具**。

### 复合索引与最左前缀
`(a, b, c)` 复合索引，查询用 `a` 或 `a,b` 或 `a,b,c` 才能命中；只用 `b` 或 `c` **命中不了**（最左前缀原则）。

## 代码示例

```sql
-- 1) 建索引
CREATE INDEX idx_users_email ON users(email);                  -- B-tree 默认
CREATE INDEX idx_events_meta ON events USING GIN (meta);       -- JSONB 用 GIN
CREATE INDEX idx_orders_user_date ON orders(user_id, created); -- 复合索引

-- 2) EXPLAIN：诊断慢查询
EXPLAIN SELECT * FROM users WHERE email = 'ada@x.com';
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 1;        -- 真实执行+耗时

-- 关键看：
--   Seq Scan（全表扫描，慢）→ 缺索引
--   Index Scan / Bitmap Index Scan（用了索引，快）
--   注意 ANALYZE 后的 actual time、rows、loops

-- 3) 最左前缀陷阱：复合索引 (user_id, created)
SELECT * FROM orders WHERE user_id = 1;              -- ✅ 命中
SELECT * FROM orders WHERE user_id = 1 AND created > '2026-01-01';  -- ✅ 命中
SELECT * FROM orders WHERE created > '2026-01-01';   -- ❌ 命中不了（缺最左 user_id）

-- 4) 部分索引：只给符合条件的行建索引，省空间、更快
CREATE INDEX idx_active_users ON users(name) WHERE deleted_at IS NULL;

-- 5) 表达式索引：对函数结果建索引（否则函数查询不走索引）
CREATE INDEX idx_users_lower_email ON users(LOWER(email));
SELECT * FROM users WHERE LOWER(email) = 'ada@x.com';  -- 现在能用索引了

-- 6) 看索引、看大小
-- \d users                 （psql 里看表的索引）
SELECT relname, pg_size_pretty(pg_relation_size(oid)) FROM pg_class
WHERE relname = 'idx_users_email';

-- 7) 删除无用索引
DROP INDEX idx_unused;
```

## ⚠️ 易错点 / 面试要点

- **「建了索引为什么没用？」**：函数包裹列（`WHERE LOWER(x)`）、隐式类型转换、`LIKE '%xx'`（前导通配符）、`OR` 条件、不符最左前缀——都可能导致索引失效。用 `EXPLAIN` 诊断。
- **`LIKE 'keyword%'` 走索引，`LIKE '%keyword'` 不走**：前缀匹配能用 B-tree，前导通配符只能全表扫描。
- **不要过度索引**：每个索引都拖慢 `INSERT/UPDATE/DELETE`（要维护索引）。只建查询真正需要的。定期用 `pg_stat_user_indexes` 找没用过的索引删掉。
- **复合索引顺序很重要**：把**选择性高（过滤性强）的列放前面**，且考虑查询是否覆盖最左前缀。
- **覆盖索引**：如果索引包含查询所需的所有列，可「仅用索引完成查询」（Index Only Scan），更快。
- **生产建索引注意锁**：普通 `CREATE INDEX` 会锁表写。大表用 `CREATE INDEX CONCURRENTLY`（不锁写，但慢且不能在事务里）。
- **EXPLAIN ANALYZE 有副作用**：它会**真正执行**（INSERT/UPDATE/DELETE 会改数据！）。分析写操作用 `EXPLAIN`（不执行）或滚在事务里回滚。
