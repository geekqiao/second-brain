# 向量数据库

> [!info] 所属：[[数据库 MOC]] · 上一篇 [[NoSQL 数据库]]

## 核心概念

**向量数据库**存储和检索**向量（embedding）**——把文本/图像/音频映射成高维数值向量，让机器能算「语义相似度」。它是**语义搜索、推荐、RAG（检索增强生成）**的核心基础设施，也是 AI 时代最热门的数据库形态。

### Embedding（嵌入）
用模型把任意内容编码成定长向量（如 768/1536 维），**语义相近的内容向量也相近**。
- 来源：用 embedding 模型（如 OpenAI text-embedding、bge、sentence-transformers）或 [[PyTorch MOC]] 自训练。
- 直觉：「猫」和「小猫」的向量距离很近，和「汽车」很远。

### 相似度度量
衡量两个向量有多近：
- **余弦相似度 Cosine**：方向夹角，最常用（关注语义方向，不关心模长）。
- **欧氏距离 L2**：直线距离。
- **内积 Inner Product**：点积（向量归一化后等价于余弦）。

### ANN（近似最近邻）—— 为什么需要专门向量库
精确找最相似的向量是 O(n)（和所有向量比一遍），千万级数据扛不住。向量库用**近似最近邻（ANN）**算法，**牺牲一点精度换巨大速度**：
- **HNSW**（分层小世界图）：查询快、召回高，主流选择，但内存大。
- **IVF**（倒排文件）：先聚类再查，内存友好。

### 向量库选型
| 方案 | 特点 | 适合 |
| --- | --- | --- |
| **pgvector**（PostgreSQL 扩展） | 复用 PG，向量+关系混合查询 | 已有 PG、中小规模、要和业务数据 JOIN |
| **Milvus** | 专用分布式向量库，可扩到亿级 | 大规模生产 |
| **Qdrant** | Rust 写，高性能，过滤强 | 中大规模、需元数据过滤 |
| **Chroma** | 轻量，Python 友好 | 原型、本地 RAG |
| **Pinecone** | 全托管云服务 | 不想运维 |

> **pgvector 是隐藏王者**：如果你已经在用 PostgreSQL，加个扩展就能做向量检索，还能把「向量相似」和「关系过滤」（如「我公司的文档」）用一句 SQL 结合——这是专用向量库不易做到的。

### RAG（检索增强生成）
向量库最火的应用：把知识库切块 → embed 入库 → 用户提问 → 把**最相关的几段**检索出来 → 塞进 LLM 提示词 → LLM 基于检索内容回答。解决 LLM「知识过时、幻觉、私有数据不懂」的问题。

## 代码示例

```python
# 1) 用 embedding 模型把文本转向量（这里用 sentence-transformers）
from sentence_transformers import SentenceTransformer
import numpy as np

model = SentenceTransformer('all-MiniLM-L6-v2')   # 384 维
docs = ["我喜欢猫", "小猫很可爱", "今天股票大跌"]
vectors = model.encode(docs)                       # → (3, 384) numpy 数组

query_vec = model.encode(["哺乳动物宠物"])           # 查询向量
# 余弦相似度找最近（语义上最接近「我喜欢猫」「小猫很可爱」）
from numpy.linalg import norm
sims = vectors @ query_vec[0] / (norm(vectors, axis=1) * norm(query_vec[0]))
print(sims)   # 越大越相似
```

```sql
-- 2) pgvector：在 PostgreSQL 里做向量检索（需 CREATE EXTENSION vector）
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE docs (
    id       SERIAL PRIMARY KEY,
    content  TEXT,
    owner_id INTEGER,
    embedding VECTOR(384)          -- pgvector 类型
);

-- 插入（应用层先算好 embedding 再写入）
INSERT INTO docs (content, owner_id, embedding)
VALUES ('我喜欢猫', 1, '[0.12, -0.03, ...]');

-- 建 HNSW 索引（ANN，加速相似查询）
CREATE INDEX ON docs USING hnsw (embedding vector_cosine_ops);

-- 检索：找与 query_vec 最相似的 5 条，且限定我的文档（关系过滤+向量！）
SELECT content, 1 - (embedding <=> '[0.1,...]'::vector) AS similarity
FROM docs
WHERE owner_id = 1                  -- ← pgvector 独有优势：向量+关系混合
ORDER BY embedding <=> '[0.1,...]'::vector   -- <=> 余弦距离，越小越近
LIMIT 5;
```

```python
# 3) RAG 最小流水线（伪代码）
# 1. 离线：知识库切块 → embed → 存入向量库
# 2. 在线：用户提问 → embed → 向量库检索 top-k 相关段 → 拼进 LLM 提示词
question = "公司的报销流程是什么？"
q_vec = model.encode([question])
hits = vector_db.search(q_vec, top_k=3, filter={"department": "finance"})
context = "\n".join(h['content'] for h in hits)
prompt = f"根据以下资料回答：\n{context}\n\n问题：{question}"

```

## ⚠️ 易错点 / 面试要点

- **向量库查的是「语义相似」不是「精确匹配」**：它返回「最相近的」，不保证精确。要精确条件（分类、归属）还得靠关系过滤——这正是 pgvector 的优势（向量+WHERE 结合）。
- **embedding 模型要与检索时一致**：存入用什么模型编码，查询就得用同一个模型，否则维度/语义空间不一致，结果无意义。换模型要全部重新 embed。
- **切块（chunking）影响 RAG 质量**：文档切太大塞不进上下文、噪声多；切太小丢上下文。常见 256~1024 token，有重叠。这是 RAG 调优关键。
- **ANN 是近似，会漏召回**：精度（ef_search/probes 参数）和速度要权衡。要求高召回调大搜索参数，但变慢。
- **要不要独立向量库？** 数据量小、要和业务数据混合查询 → pgvector 就够（一套库、一句 SQL）。海量向量、纯检索、要独立扩展 → Milvus/Qdrant。别为了「用向量库」而多维护一套系统。
- **向量索引的选择**：HNSW 查询快召回高但吃内存；IVF 内存省但需训练聚类。pgvector 两者都支持。
- **维度与成本**：向量维度越高表达力越强但存储/计算越贵。常见 384/768/1536 维，按模型定。
