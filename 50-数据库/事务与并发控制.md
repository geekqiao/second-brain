# 事务与并发控制

> [!info] 所属：[[数据库 MOC]] · 上一篇 [[索引与查询优化]] · 下一篇 [[NoSQL 数据库]]

## 核心概念

多个用户同时改数据库时，如何保证**数据正确**？答案就是**事务（Transaction）**。它是数据库最重要的特性之一，理解 ACID、隔离级别、MVCC，才能写出在高并发下不出错的应用。

### 事务与 ACID
事务是把多个操作打包成「**要么全成功、要么全回滚**」的执行单元。四个保证：
- **A**tomicity 原子性：全部成功或全部回滚，不存在中间态。
- **C**onsistency 一致性：事务前后数据满足约束（从一个正确状态到另一个）。
- **I**solation 隔离性：并发事务互不干扰（程度由隔离级别定）。
- **D**urability 持久性：提交后即使断电也不丢（靠 WAL 日志）。

### 事务语法
```sql
BEGIN;        -- 开启
-- SQL ...
COMMIT;       -- 提交（生效）/ ROLLBACK; 回滚（撤销）
```

### 并发的三种「脏」现象
| 现象 | 含义 |
| --- | --- |
| **脏读** Dirty Read | 读到别的事务**未提交**的数据（它可能回滚） |
| **不可重复读** Non-repeatable Read | 同一查询读两次，**同一行**值变了（别人改了并提交） |
| **幻读** Phantom Read | 同一查询读两次，**行数**变了（别人插入/删除了行） |

### 四个隔离级别（隔离强度↑，性能↓）
| 级别 | 脏读 | 不可重复读 | 幻读 |
| --- | --- | --- | --- |
| Read Uncommitted | 可能 | 可能 | 可能 |
| **Read Committed**（PG 默认） | 杜绝 | 可能 | 可能 |
| Repeatable Read | 杜绝 | 杜绝 | 可能（PG 在此级也杜绝幻读） |
| Serializable | 杜绝 | 杜绝 | 杜绝（最强，可能串行化失败需重试） |

> 取舍：隔离越强越安全，但并发性能越低。多数应用用默认（Read Committed）。

### MVCC（多版本并发控制）
PostgreSQL（及现代 DB）的核心并发机制：**修改不覆盖旧数据，而是产生新版本**。
- **读操作读旧版本快照**，**写操作创建新版本** → 读不阻塞写、写不阻塞读！
- 代价：旧版本需定期清理（`VACUUM`），否则「膨胀」。

### 锁与 SELECT FOR UPDATE
显式加行锁，防止并发修改同一条：
- `SELECT ... FOR UPDATE`：锁定行，事务结束才释放（用于「读取-修改-写回」场景防覆盖）。

## 代码示例

```sql
-- 1) 经典事务：转账（扣款 + 加款，要么都成要么都回滚）
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;   -- A 扣 100
UPDATE accounts SET balance = balance + 100 WHERE id = 2;   -- B 加 100
-- 若任一步失败，整体回滚，不会出现「扣了没加」
COMMIT;
-- 出错时：ROLLBACK;

-- 2) 设置隔离级别（在 BEGIN 后、第一条语句前）
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE id = 1;   -- 第一次读
-- （别处提交了修改）
SELECT balance FROM accounts WHERE id = 1;   -- Repeatable Read 下值不变
COMMIT;

-- 3) SELECT FOR UPDATE：防并发覆盖（「读-改-写」经典坑）
-- 场景：两人同时抢最后一件库存，不加锁会超卖
BEGIN;
SELECT stock FROM products WHERE id = 42 FOR UPDATE;   -- 锁住这行
-- 应用判断 stock > 0
UPDATE products SET stock = stock - 1 WHERE id = 42;
COMMIT;   -- 提交后才释放锁，另一人必须等

-- 4) 乐观锁（另一种思路）：用 version 字段，更新时检查
UPDATE products SET stock = stock - 1, version = version + 1
WHERE id = 42 AND version = 7;   -- 若 version 已被别人改，影响行数=0，应用重试

-- 5) 死锁演示：两事务反向锁对方要的行 → 死锁，PG 自动检测并 abort 一个
-- T1: UPDATE accounts SET ... WHERE id=1; 然后 id=2
-- T2: UPDATE accounts SET ... WHERE id=2; 然后 id=1   ← 死锁
```

## ⚠️ 易错点 / 面试要点

- **「读-改-写」竞态**：先 `SELECT` 判断再 `UPDATE`，中间无锁的话，并发会出问题（超卖、重复操作）。解法：`SELECT FOR UPDATE`（悲观）或 `version` 字段（乐观）。
- **隔离级别默认值**：PostgreSQL 默认 **Read Committed**，MySQL InnoDB 默认 **Repeatable Read**。面试常考差异。
- **MVCC 的代价是膨胀**：频繁更新产生大量旧版本，需 `VACUUM`（或自动 `autovacuum`）回收，否则表和索引膨胀、查询变慢。这是 PG 运维重点。
- **长事务危害大**：长事务持有快照，阻塞 `VACUUM` 清理旧版本（导致膨胀），还持锁阻塞别人。生产要监控、避免长事务。
- **死锁**：两个事务互相等对方持有的锁。PG 会自动检测并杀掉一个（报错），应用要能**捕获并重试**。规范加锁顺序可预防。
- **事务要短**：事务越长，持锁越久、并发越差。不要在事务里做耗时操作（如调外部 API）。
- **嵌套事务 / SAVEPOINT**：事务里用 `SAVEPOINT` 设保存点，部分失败可回滚到保存点而非整个事务。
