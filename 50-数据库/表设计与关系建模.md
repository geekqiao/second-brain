# 表设计与关系建模

> [!info] 所属：[[数据库 MOC]] · 上一篇 [[PostgreSQL 入门与核心特性]] · 下一篇 [[索引与查询优化]]

## 核心概念

**好的表设计**决定应用的命运——设计乱，查询难写、数据冗余、更新异常；设计好，清晰、高效、易维护。核心工具是**范式（Normalization）**与**关系建模**。

### 范式（Normalization）—— 消除冗余与异常
把数据组织得「每张表只表达一件事」。常用三级：
- **1NF**：每列不可再分（原子值），无重复组。
- **2NF**：在 1NF 基础上，非主键列**完全依赖**主键（消除部分依赖）。
- **3NF**：在 2NF 基础上，非主键列**直接依赖**主键（消除传递依赖）。

> 口诀：**「每个非键字段只依赖主键、且只依赖整个主键、不依赖其它非键字段」**。达到 3NF 一般够用。

### 反范式（Denormalization）
有时为了**查询性能**故意违反范式（加冗余字段、避免频繁 JOIN）。这是取舍：以冗余换速度，需保证一致性。高并发读场景常见。

### 三种关系建模
| 关系 | 建模方式 | 示例 |
| --- | --- | --- |
| **1:1** | 一方加唯一外键 | 用户 ↔ 用户详情 |
| **1:N** | 「N」方加外键指向「1」方 | 部门(1) ↔ 员工(N) |
| **N:M** | 建**中间表**（两个外键，常组复合主键） | 学生 ↔ 课程（选课表） |

### 约束（Constraint）—— 让数据库替你保证正确
- `PRIMARY KEY`：主键，唯一非空。
- `FOREIGN KEY`：外键，保证参照完整性。
- `UNIQUE`：唯一。
- `NOT NULL`：非空。
- `CHECK`：自定义条件（如 age >= 0）。
- `DEFAULT`：默认值。

> [!tip] 设计哲学
> **把业务规则用约束写进数据库**（而非只在应用层校验）。数据库是最后一道防线，约束在并发、多入口写入时仍能保证数据正确。但别过度依赖外键做复杂逻辑（影响性能和迁移）。

## 代码示例

```sql
-- 1:N 关系：部门(1) ↔ 员工(N)
CREATE TABLE departments (
    id   SERIAL PRIMARY KEY,
    name TEXT NOT NULL UNIQUE
);
CREATE TABLE employees (
    id    SERIAL PRIMARY KEY,
    name  TEXT NOT NULL,
    dept_id INTEGER REFERENCES departments(id),   -- N 方加外键
    salary NUMERIC(10,2) CHECK (salary > 0)       -- 约束
);

-- N:M 关系：学生 ↔ 课程，用选课表
CREATE TABLE students (id SERIAL PRIMARY KEY, name TEXT);
CREATE TABLE courses  (id SERIAL PRIMARY KEY, title TEXT);
CREATE TABLE enrollments (                       -- 中间表
    student_id INTEGER REFERENCES students(id) ON DELETE CASCADE,
    course_id  INTEGER REFERENCES courses(id)  ON DELETE CASCADE,
    grade      CHAR(2),
    PRIMARY KEY (student_id, course_id)          -- 复合主键，防重复选课
);

-- 反范式示例：订单表冗余存「下单时的商品名」
-- （即使商品改名，历史订单仍显示当时名称）
CREATE TABLE orders (
    id        SERIAL PRIMARY KEY,
    product_id INTEGER REFERENCES products(id),
    product_name_snapshot TEXT NOT NULL           -- 冗余快照字段
);

-- 软删除 vs 硬删除
-- 硬删除：DELETE FROM users WHERE id=1;        （数据真没了）
-- 软删除：加 deleted_at 列，删除=置时间戳        （可恢复，但查询都要 WHERE deleted_at IS NULL）
CREATE TABLE users (
    id         SERIAL PRIMARY KEY,
    name       TEXT,
    deleted_at TIMESTAMP,                          -- NULL=未删
    CHECK (deleted_at IS NULL OR deleted_at <= now())
);
```

## ⚠️ 易错点 / 面试要点

- **范式不是越高越好**：3NF 是平衡点。盲目追高范式（BCNF/4NF）增加 JOIN、降低性能；过度反范式又冗余难维护。按场景取舍。
- **N:M 别用逗号分隔字段**：把「多个课程 ID」塞进一个字符串列（`"1,2,3"`）是经典反模式——无法 JOIN、无法索引、无法保证完整性。必须用中间表。
- **外键的取舍**：外键保证完整性，但写入有开销、迁移/分库时碍事。互联网高并发场景有时在**应用层**保证关系而禁用外键。要懂权衡。
- **选对主键**：自增 INT 简单高效但顺序暴露；UUID 全局唯一适合分布式但占空间、索引稍慢。避免用业务字段（如手机号）做主键（可能变）。
- **软删除的坑**：软删除后，唯一约束（如 email UNIQUE）会把「已删用户」的 email 占住，导致无法重新注册。需配合部分索引或设计。
- **命名规范**：表名复数或单数统一（如全用 `user` 或全用 `users`）；外键列用 `<表>_id`；时间戳用 `created_at`/`updated_at`。团队统一最重要。
- **预留审计字段**：`created_at`、`updated_at`、`created_by` 几乎每张表都该有，便于追溯。可用触发器自动维护 `updated_at`。
